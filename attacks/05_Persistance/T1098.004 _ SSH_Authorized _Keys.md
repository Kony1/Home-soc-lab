

## 🧩 MITRE ATT&CK – Persistence: SSH Authorized Keys (T1098.004)

## 🎯 Tactic: Persistence (TA0003)

## 🔧 Technique: T1098.004 – SSH Authorized Keys

## 🖥️ Platform: Linux (Ubuntu), Unix-like systems

## 📡 Detection Sources: Syslog, auth.log, auditd, Wazuh, Microsoft Sentinel, AMA


## 📝 1. Popis techniky

Útočník, který získá přístup k Linux systému, může vytvořit trvalý přístup tím, že přidá svůj veřejný SSH klíč do souboru:

~/.ssh/authorized_keys
Tím získá možnost přihlásit se bez hesla, i když:

změníš heslo uživatele

zakážeš původní přístup

resetuješ SSH konfiguraci

Tahle technika je extrémně běžná v reálných útocích, protože je:

tichá

nenápadná

snadno přehlédnutelná

velmi spolehlivá


## 🛠️ 2. Lab scénář (útok)

### 2.1 Generování útočníkova klíče

Kód
ssh-keygen -t rsa -b 4096 -f attacker_key

### 2.2 Přidání klíče do persistence

Kód
 mkdir -p ~/.ssh
echo "<PUBLIC_KEY>" >> ~/.ssh/authorized_keys

chmod 600 ~/.ssh/authorized_keys

### 2.3 Ověření přístupu

Kód
ssh -i attacker_key ubuntu@<IP>

## 🔍 3. Logy a artefakty (co budeš dohledávat)


### 🐧 Ubuntu / Linux

auth.log / syslog
změna souboru authorized_keys

nové SSH přihlášení pomocí publickey

změna práv v ~/.ssh

auditd (pokud povoleno)
Pravidlo:

Kód
-w /home/ubuntu/.ssh/authorized_keys -p wa -k ssh-persistence

#### Logy:

Kód
ausearch -k ssh-persistence
Microsoft Sentinel (AMA + Syslog DCR)
Detekce změny authorized_keys
Kód
#### Syslog
| where Message contains "authorized_keys"
Detekce SSH loginu pomocí klíče
Kód

#### Syslog


| where ProcessName == "sshd"

| where Message contains "Accepted publickey"


### 🪟 Windows 11 / Windows Server

Windows nemá authorized_keys, ale pokud:

máš nainstalovaný OpenSSH Server

útočník přidá klíč do:
C:\ProgramData\ssh\administrators_authorized_keys  
nebo
C:\Users\<user>\.ssh\authorized_keys

→ logy najdeš v Event Viewer → OpenSSH/Operational.

Hledej:

Event ID 14 – AuthorizedKeys file modified

Event ID 3 – SSH login using key

### 🛡️ Wazuh

Wazuh má předpřipravené detekce:

změna authorized_keys

změna práv v ~/.ssh

podezřelé SSH přihlášení

Hledej v:

Kód
/var/ossec/logs/alerts/alerts.json
Typické rule IDs:

5710 – SSH authorized_keys modified

5712 – SSH directory modified

5716 – SSH publickey authentication


### ☁️ Microsoft Sentinel

Pokud máš:

AMA agenta

Syslog DCR

Wazuh connector (optional)

→ uvidíš logy v:

Syslog

CommonSecurityLog

#### Wazuh_CL (pokud používáš Wazuh → Sentinel)



## 🧠 4. Detekce (SOC pohled)

#### 🔎 KQL – změna authorized_keys
Kód
Syslog
| where Message contains "authorized_keys"
| project TimeGenerated, HostName, ProcessName, Message


#### 🔎 KQL – SSH login pomocí klíče

Kód
Syslog
| where Message contains "Accepted publickey"
| project TimeGenerated, HostName, User, Message

#### 🔎 KQL – Wazuh alert
Kód
Wazuh_CL
| where rule_id in ("5710","5712","5716")


##🛡️ 5. Hardening / Remediace

odstranit neznámé klíče z authorized_keys

nastavit správná práva:

Kód
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
povolit MFA (FIDO2 keys)

zakázat password login:

Kód
PasswordAuthentication no
monitorovat změny v ~/.ssh pomocí auditd nebo Wazuh
